Record an analysis of enq: TX - row lock contention

2018-03-23 ​​04:18 by AlfredZhao, 61 Readings, 0 Comments, Favorites , Editing

Fault description: communicate with customers, preliminary confirmation of the fault range is probably between 8:30-10:30 am, the fault phenomenon is the number of Tomcat connection is full and the application can not be connected, the database alert no obvious error, need to assist in the investigation the reason.

  • 1. Import data containing fault moments
  • 2 create m_ash table, clear fault time

1. Import data containing fault moments

In order to facilitate the follow-up analysis, I asked the customer for awrdump from 13:00 yesterday to 18:00 today to import into his own experimental environment for analysis.

The production environment exports awrdump:

@?/rdbms/admin/awrextr

Test environment import awrdump:

SYS@jyzhao1 >select * from dba_directories;
SYS@jyzhao1 >create directory jy as '/home/oracle/awrdump'; 
SYS@jyzhao1 >select * from dba_directories;
SYS@jyzhao1 >!mkdir -p /home/oracle/awrdump

SYS@jyzhao1 >@?/rdbms/admin/awrload

End of AWR Load

2 create m_ash table, clear fault time

Create m_ash table:

--create table 
create table m_ash20180322 as select * from dba_hist_active_sess_history where dbid=&dbid;

Enter the dbid corresponding to the production database and create the analysis table.

select to_char(sample_time, 'yyyy-mm-dd hh24:mi'), count(1)
  FROM m_ash20180322
 group by to_char(sample_time, 'yyyy-mm-dd hh24:mi')
 order by 1;

Generate a line chart based on the generated data as follows:

From the figure, it can be clear that the fault time, ie, the sessions at 10:00, 12:30, and 14:10, has increased significantly (backlog). It appears that the customer's feedback time point does not include all abnormal times.

In addition, refer to the maclean diagnostic script, you can see the core meaning is almost the same, but further separation of the instance_number:

--验证导出的ASH时间范围:
select
 t.dbid, t.instance_number, min(sample_time), max(sample_time), count(*) session_count
  from m_ash20180322 t
 group by t.dbid, t.instance_number
 order by dbid, instance_number;
 
--确认问题发生的精确时间范围:
 select 
 dbid, instance_number, sample_id, sample_time, count(*) session_count
  from m_ash20180322 t
 group by dbid, instance_number, sample_id, sample_time
 order by dbid, instance_number, sample_time;

3. Determine the top n event of the abnormal moment

Determine the top n event for each sample point. The following is also a reference to maclean's script.
For example, here I take 2018-03-22 09:59:00 - 2018-03-22 10:00:00 as an example:

select t.dbid,
       t.sample_id,
       t.sample_time,
       t.instance_number,
       t.event,
       t.session_state,
       t.c session_count
  from (select t.*,
               rank() over(partition by dbid, instance_number, sample_time order by c desc) r
          from (select /*+ parallel 8 */
                 t.*,
                 count(*) over(partition by dbid, instance_number, sample_time, event) c,
                 row_number() over(partition by dbid, instance_number, sample_time, event order by 1) r1
                  from dba_hist_active_sess_history t
                where sample_time >
                    to_timestamp('2018-03-22 09:59:00',
                                 'yyyy-mm-dd hh24:mi:ss')
                and sample_time <
                    to_timestamp('2018-03-22 10:00:00',
                                 'yyyy-mm-dd hh24:mi:ss')
                ) t
         where r1 = 1) t
 where r < 3
 order by dbid, instance_number, sample_time, r;

For other abnormal moments, enter the corresponding variable value:

4. Determine the final top holder

Using maclean's script, observe the wait chain for each sample point:

select 
 level                     lv,
 connect_by_isleaf         isleaf,
 connect_by_iscycle        iscycle,
 t.dbid,
 t.sample_id,
 t.sample_time,
 t.instance_number,
 t.session_id,
 t.sql_id,
 t.session_type,
 t.event,
 t.session_state,
 t.blocking_inst_id,
 t.blocking_session,
 t.blocking_session_status
  from m_ash20180322 t
where sample_time >
    to_timestamp('2018-03-22 09:59:00',
                 'yyyy-mm-dd hh24:mi:ss')
and sample_time <
    to_timestamp('2018-03-22 10:00:00',
                 'yyyy-mm-dd hh24:mi:ss')
 start with blocking_session is not null
connect by nocycle
 prior dbid = dbid
       and prior sample_time = sample_time
          /*and ((prior sample_time) - sample_time between interval '-1'
          second and interval '1' second)*/
       and prior blocking_inst_id = instance_number
       and prior blocking_session = session_id
       and prior blocking_session_serial# = session_serial#
 order siblings by dbid, sample_time;

5. Summary

From the fourth step, it can be seen that the top holder is the instance 1 and the session 3548. For
example, it can be seen that the 481 session of the instance 1 is blocked by the 6377 session of the instance 2, and then the 6377 session of the instance 2 is blocked by the 3548 session of the instance 1.
The sql text can be queried by sql_id:

select * from dba_hist_sqltext where sql_id = '&sql_id';

It can be seen that the SQL currently executing in the 3548 session of Example 1 is just a query statement. The current session state is ON CPU, so it is inferred that the DML transaction before the session has not been submitted results in blocking.
When querying the DML operation of the session, there are also update and insert operations, but the update operation cannot find the corresponding SQL text.

select t.event, t.*
  from m_ash20180322 t
 where instance_number = 1
   and session_id = 3548
   and t.sql_opname <> 'SELECT';

In fact, from the ash can also see the information on the 3548 blocking, and even from the addm's proposal will have similar suggestions:

   Rationale
      The session with ID 3548 and serial number 8795 in instance number 1 was
      the blocking session responsible for 52% of this recommendation's
      benefit.
   Rationale
      The session with ID 6377 and serial number 30023 in instance number 2
      was the blocking session responsible for 47% of this recommendation's
      benefit.

Only we query from the bottom, you can see 6377 is actually blocked by 3548, find the final blocker.
